# Content Mediatheque Plan

Статус: planned UX/product track after Media Intelligence validation.

## Цель

Добавить в раздел `Контент` отдельное подменю `Медиатека`, где владелец видит фото бизнеса как готовую библиотеку для публикаций, карт и соцсетей.

Пользовательская идея:

```text
Фото уже разобраны.
LocalOS понимает, что подходит для постов и карт.
Понятно, чего не хватает.
```

## Information Architecture

```text
Контент
  Календарь
  Медиатека
```

`Календарь` остаётся главным рабочим экраном публикаций.

`Медиатека` становится местом для загрузки, проверки и выбора фото. Это не файловый менеджер и не технический asset manager.

## Photo Card

Карточка фото показывает:

- превью;
- категорию: `вход`, `интерьер`, `процесс`, `результат`, `команда`, `атмосфера`;
- качество: `подходит`, `лучше заменить`, `нужно проверить`;
- где использовать: `Яндекс`, `2ГИС`, `Google`, `Instagram`, `Telegram`, `VK`;
- когда использовалось последний раз.

Не показывать пользователю:

- `asset_id`;
- `provider`;
- `Vision API`;
- `prompt_hash`;
- технический cache.

## Empty State

Эмоция:

```text
“Я понимаю, какие фото нужны бизнесу.”
```

Показывать пример заполненной медиатеки и CTA:

```text
Загрузить первые фото
```

Текст:

```text
LocalOS разберёт фото и подскажет, что использовать в публикациях и на картах.
```

## Upload Flow

1. Пользователь нажимает `Загрузить фото`.
2. Видит, что интеллектуальная работа с фотографиями расходует кредиты.
3. Подтверждает загрузку.
4. Фото попадают в медиатеку.
5. LocalOS анализирует каждое новое фото один раз.
6. Повторное использование фото не списывает кредиты.

## Publication Drawer Integration

В drawer публикации блок `Фото` остаётся простым:

```text
Фото

Выбрано лучшее фото
[preview]

Подходит для Telegram и VK.
Для Яндекс лучше добавить фото входа.

[Заменить] [Загрузить новое]
```

Если подходящего фото нет:

```text
Фото не выбрано

Для этой публикации лучше снять:
готовую причёску ребёнка или процесс стрижки.

[Загрузить фото]
```

## Backend/API Requirements

Использовать существующий foundation:

- `photo_assets`;
- `ai_runtime_cache`;
- `ai_usage_events`;
- `photo_asset_usage_events`;
- `vision_enabled`.

Нужны следующие product endpoints:

- upload original photo to S3-compatible storage;
- create `photo_asset`;
- create preview/thumb versions;
- analyze only new `asset_version`;
- list media library with filters;
- mark usage from publication;
- replace/reanalyze photo version.

## Acceptance Criteria

- В `Контенте` есть подменю `Медиатека`.
- Пользователь видит покрытие фото задач бизнеса.
- Пользователь видит, каких фото не хватает.
- Фото можно загрузить без технических терминов.
- Новое фото анализируется один раз.
- Повторное использование фото не списывает кредиты.
- В публикации LocalOS предлагает лучшее фото или объясняет, что снять.
